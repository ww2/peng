# PLAN: Non-regular AFC field

## Approach

Keep `#manual-afc` as the **total** monthly AFC (unchanged semantics, unchanged pension formula). Add a new sibling field `#nonreg-afc` that holds the **non-regular portion** of that total. Visible only on `mode: 'total'` plans (`hybrid-pre2012`, `contributory-pre2012`, `noncontributory`). Optional, defaults to 0. Lays groundwork for re-enabling raises later (raises will blend into `total − nonRegular` only).

Settled from the conversation:
- Hide field entirely on `mode: 'regular'` plans.
- DP solver populates **both** fields (total → existing, non-regular sum → new).
- New URL param for the field, named `nonRegularAfc`.
- Raise re-enablement is a later, separate task.
- `canCalculate()` requires `total > 0`; non-regular optional, defaults 0.
- Dual-method (pre-1971) must also yield a regular/non-regular split for the winning method, since future raises apply only to the regular component.
- **Layout & labels.** Add a header row above the existing `#manual-afc` input containing the label `Regular`. When the new field is visible (only on `mode: 'total'` plans), the header `Non-regular` sits to the right of `Regular` on that header row, and the new input sits to the right of `#manual-afc` on the input row. The existing `form-label` for the row stays as `Monthly AFC ($)`. Implementation cue: same shape as the credited-service `svc-pair-row` at `index.html:352-373`, with a `Regular` pair always shown and a `Non-regular` pair toggled by plan mode.
- **Persistence on plan switch.** Persist the non-regular value when switching to a `mode: 'regular'` plan (just hide the input; don't clear it). The URL param `nonRegularAfc` is also kept on those plans — plans that ignore it ignore it harmlessly.
- **Validation.** Hard block when `nonRegular > total`: apply `input-error` to `#nonreg-afc` and have `canCalculate()` return false until corrected. (Equality is allowed — all earnings being non-regular is theoretically valid.)
- **DP-computed summary display.** Augment the existing `afc-computed` line inline. Format becomes `Best N windows (<mode> pay): $X/mo (includes $NR non-regular pay)`. The `regular-earnings` line is unchanged.
- **Dual-method (pre-1971) display.** Show the same `(includes $NR non-regular pay)` parenthetical on both the winner and runner-up lines. Requires threading the runner-up's `result.selected` through `computeDualMethodAfc`'s return so its non-regular sum can be computed.
- **Parenthetical when `nonReg = 0`.** Suppress — don't render `(includes $0 non-regular pay)`. (Assumption; revise during stage review if undesired.)
- **Confirm-dialog wording.** Unchanged.
